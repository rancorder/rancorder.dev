## なぜこのシステムが必要だったのか

私は17年間、製造業でエンタープライズPMとして「技術的に完成したPoC」を「安定稼働するProduction」に移行させる仕事をしてきた。そこで学んだ最も重要な原則は、**「個人の努力に依存しない構造を作る」**ということだ。

### コンテンツ発信の構造的課題

技術ブログを書く人間は、こんな課題に直面する：

- **プラットフォームが分散**：Qiita、Zenn、個人ブログ、note...どこに書くべきか？
- **転載の手間**：同じ内容を複数サイトにコピペする虚しさ
- **SEO効果の分散**：コンテンツが散らばり、検索流入が分散
- **運用負荷**：サイトマップ更新、インデックス管理、デザイン調整...

## 採用した技術スタックとその理由

30時間という限られた時間で、最大の価値を生み出すための技術選定。

</FadeIn>

<FadeIn delay={2200}>

### Next.js 14 (App Router)

**選定理由**：
- Server/Client Componentsの明確な分離
- ファイルベースルーティング（`/blog/[slug]`が自動生成）
- ISR（Incremental Static Regeneration）によるパフォーマンス最適化
- Vercelとの完璧な統合

<CalloutBox type="success" title="重要な判断">
Pages RouterではなくApp Routerを採用。Server Componentsを最大限活用することで、クライアントサイドの負荷を最小化。
</CalloutBox>

</FadeIn>

<FadeIn delay={2400}>

### MDX（Markdown + JSX）

**選定理由**：
- Markdownの書きやすさ + Reactコンポーネントの表現力
- Frontmatter（メタデータ）のサポート
- TypeScript型チェックの恩恵

### Prebuild Cache System

**課題**：Vercelのビルド時は外部API（Qiita、Zenn）へのアクセスが制限される

**解決策**：prebuildスクリプトで事前に記事を取得し、JSONキャッシュとして保存

```javascript
// package.json
{
  "scripts": {
    "prebuild": "node scripts/fetch-external-articles.js",
    "build": "next build"
  }
}
```

**フロー**：
1. `prebuild`実行 → Qiita/ZennのAPIから記事取得
2. `public/external-articles.json`に保存
3. `build`実行 → キャッシュを読み込んでサイトマップ生成

</FadeIn>

<FadeIn delay={2800}>

<CalloutBox type="info" title="技術的ポイント">
Vercelのビルド時制約を「prebuildフック」で回避。これにより、外部プラットフォームの記事も完全にインデックス可能に。
</CalloutBox>

</FadeIn>

<FadeIn delay={3000}>

### TypeScript + CSS Modules

**選定理由**：
- 型安全性（コンパイル時エラー検出）
- CSS Modulesでスタイルの衝突回避
- IDEの補完サポート

**失敗から学んだこと**：
- ❌ styled-jsxはServer Componentsで使えない
- ❌ CSS Modulesでグローバルセレクタは使えない
- ✅ グローバルスタイルは`globals.css`、コンポーネントスタイルはCSS Modules

</FadeIn>

### 1. 自動インデックスシステム

```typescript
// lib/mdx.ts（簡略版）
export async function getAllPosts() {
  const files = fs.readdirSync(BLOG_DIR);
  const posts = files
    .filter(file => file.endsWith('.mdx'))
    .map(file => {
      const content = fs.readFileSync(path.join(BLOG_DIR, file));
      const { data } = matter(content); // Frontmatter解析
      return {
        slug: file.replace('.mdx', ''),
        ...data
      };
    });
  return posts;
}
```

**仕組み**：
- `content/blog/`にMDXファイルを置く
- ファイル名が自動的にURLスラッグになる
- Frontmatterからメタデータを抽出
- 読了時間を自動計算

</FadeIn>

<FadeIn delay={3600}>

### 2. 外部記事統合（Qiita/Zenn）

```javascript
// scripts/fetch-external-articles.js（概要）
async function fetchExternalArticles() {
  // Qiitaから取得
  const qiitaResponse = await fetch('https://qiita.com/api/v2/users/rancorder/items');
  const qiitaArticles = await qiitaResponse.json();
  
  // Zennから取得
  const zennResponse = await fetch('https://zenn.dev/api/articles?username=supermassu');
  const zennArticles = await zennResponse.json();
  
  // 統合して保存
  const allArticles = [...qiitaArticles, ...zennArticles];
  fs.writeFileSync('public/external-articles.json', JSON.stringify(allArticles));
}
```

**結果**：
- 内部記事：2件
- Qiita記事：3件
- Zenn記事：4件
- **合計：9件の記事を一元管理**

### 3. サイトマップ自動生成

```typescript
// app/sitemap.ts（簡略版）
export default async function sitemap() {
  const posts = await getAllPosts(); // 内部記事
  const external = await getExternalArticles(); // 外部記事
  
  return [
    { url: 'https://rancorder.vercel.app/', lastModified: new Date() },
    { url: 'https://rancorder.vercel.app/blog', lastModified: new Date() },
    ...posts.map(post => ({
      url: `https://rancorder.vercel.app/blog/${post.slug}`,
      lastModified: new Date(post.date)
    })),
    ...external.map(article => ({
      url: article.link,
      lastModified: new Date(article.date)
    }))
  ];
}
```

**効果**：
- Google Search Consoleに自動送信
- 11ページを完全インデックス
- SEO効果の最大化
